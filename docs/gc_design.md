# Hxcpp GC 架构升级设计

## 1. 概述
本文档概述了 hxcpp 垃圾回收器 (Immix) 的升级设计，旨在支持：
1.  **多线程并行清扫 (Multithreaded Parallel Sweep)**：以提高吞吐量并减少暂停时间。
2.  **后台增量整理 (Background Incremental Compact)**：通过将整理阶段分解为更小的、与业务线程交替执行的增量步骤，来减少最大暂停时间。

## 2. 多线程清扫 (Multithreaded Sweep)

### 2.1. 目标
将清扫吞吐量提高 >= 30%，并将暂停时间减少 >= 20%。

### 2.2. 架构
*   **工作线程 (Worker Threads)**：GC 工作线程池（默认为核心数或其一部分）。
*   **任务队列 (Task Queue)**：共享的“清扫任务”队列。
*   **任务粒度 (Task Granularity)**：一个“块组 (Block Group)”或一系列块。`Immix` 以 32KB 的块（或类似大小）分配内存。`GlobalAllocator` 维护已分配块的列表。我们可以将此列表划分为多个块（Chunks）。
*   **同步 (Synchronization)**：
    *   `GlobalAllocator::reclaimBlocks` 将不再线性迭代。
    *   它会将 `mRowBlocks` 和 `mRecyclableBlocks` 划分为多个块，并将它们推送到 `GCTaskQueue`。
    *   工作线程提取块并执行 `BlockDataInfo::reclaim` 逻辑。
    *   **位图同步 (Bitmap Synchronization)**：`BlockDataInfo::reclaim` 修改 `mRowMarked` 和 `allocStart`。这些是每个块独有的，因此只要我们不在块列表本身上发生竞争，并行处理*不同*的块就是安全的，无需加锁。
    *   **空闲列表更新 (Free List Update)**：回收的块被添加到 `mFreeBlocks`。这是一个共享列表。我们将使用 TLAB（线程本地分配缓冲区）风格的方法：每个工作线程在本地列表中收集回收的块，并在结束时（或分批）使用锁将其合并到全局空闲列表。

### 2.3. 数据结构
*   `struct SweepTask { BlockDataInfo** start; size_t count; };`
*   `class GCThreadPool`: 管理 `HxCreateDetachedThread` 工作线程和一个用于信号通知的 `HxSemaphore`。

## 3. 后台增量整理 (Background Incremental Compact)

### 3.1. 目标
将最大暂停时间减少到 < 10ms (p99)。

### 3.2. 架构
当前的 `MoveBlocks` 是一个单体式的 Stop-The-World (STW) 操作。我们将把它重构为一个状态机：

*   **状态机 (State Machine)**：
    *   `IDLE`: 正常执行。
    *   `MARKING`:（现有逻辑，暂时不变，假定为 STW 或并发标记（如果已支持））。
    *   `COMPACT_PREPARE`: 识别碎片，计算移动计划。（STW，快速）。
    *   `COMPACT_EVACUATE_STEP`: 移动一批对象。（STW，< 5ms）。
    *   `COMPACT_UPDATE_POINTERS_STEP`: 更新指向已移动对象的指针。（STW，< 5ms）。
    *   `COMPACT_RECLAIM`: 回收旧块。（STW，快速）。

*   **增量逻辑 (Incremental Logic)**：
    *   我们不再一次性移动*所有*碎片块，而是选择一个候选集（例如 10 个块）。
    *   **步骤 1 (Evacuate)**：将对象从这 10 个块移动到新块。记录转发信息。
    *   **步骤 2 (Update)**：扫描根和堆以更新指针。
        *   *挑战*：仅为了 10 个块而扫描*整个*堆来更新指针是低效的（O(Heap)）。
        *   *解决方案*：我们必须要么整理*足够多*的块以证明扫描的合理性，要么使用记忆集/卡表（如果可用）。Immix 是基于区域的收集器。
        *   *修正方法*：如果没有针对*每个*对象的记忆集，我们就无法轻易做到“部分指针更新”。
        *   **替代方案**：需求要求“后台增量... 4 步...”。“疏散计划... 对象复制... 指针更新... 空间回收”。
        *   我们将实现一个 **有界整理 (Bounded Compaction)**：限制每个 GC 周期移动的块数，以保持暂停时间在 10ms 以下。如果我们不能整理所有内容，我们将推迟到下一个周期。
        *   等等，用户要求“后台... 连续步骤允许业务线程运行”。
        *   这意味着我们将*整个*整理过程分解为多个步骤。
        *   为了支持增量“更新指针”，我们需要知道指针在*哪里*。`AdjustPointers` 会扫描所有内容。
        *   如果我们拆分 `AdjustPointers`，我们需要在中断处恢复。但业务线程可能会更改指针。
        *   **写屏障 (Write Barrier)**：我们需要一个能捕获指针写入的写屏障。
        *   如果业务线程写入指针 `p = old_obj`，而我们尚未将 `old_obj` 更新为 `new_obj`，就会出现问题。
        *   *染色指针 (Colored Pointers)*：我们可以使用“Mark ID”或特定位来指示“已移动”。
        *   **协议 (Protocol)**：
            1.  **Evacuate**：移动对象。旧对象标记为“MOVED”（头部的转发指针）。
            2.  **Incremental Update**：逐块扫描堆以更新指针。
            3.  **Barrier**：如果业务线程加载指针，它必须检查它是否指向 MOVED 对象（读屏障）。
                *   *问题*：hxcpp 没有读屏障。
            4.  **写屏障方法**：
                *   只有当我们有读屏障或虚拟内存技巧时，我们才支持*并发复制*。
                *   *回退*：我们将假设“增量”意味着“将工作拆分为小的 STW 块”。
                *   为了在没有读屏障的情况下增量处理“指针更新”，我们必须确保业务线程*永远不会看到*旧指针。这需要在允许业务线程运行*之前*更新*所有*根和*所有*存活的堆指针。这违背了增量更新的目的。
                *   *除非*：我们使用 **虚拟内存 (mprotect)**。保护包含旧对象的页面。如果业务线程访问它们，则捕获并修复。（太复杂/特定于平台）。
                *   *用户约束检查*：“使用写屏障和染色指针... 保证对象视图一致性”。
                *   这意味着我们*可以*检查指针。也许我们只在*写入*时检查？
                *   如果我读取 `ptr = obj->field`，并且 `obj->field` 指向 `old_loc`，而我解引用 `ptr`，我会崩溃/读取垃圾数据。
                *   所以“染色指针”必须意味着我们屏蔽指针或检查标记。
                *   **解读**：用户可能假设了一种特定的 GC 模型（如 Baker 或 Shenandoah）。
                *   **建议实现**：
                    *   我们将实现一个**带功能标志**的增量整理器。
                    *   我们将把 `AdjustPointers` 阶段分解为块。
                    *   **关键 Hack**：我们依赖于 `hxcpp` 使用 `hx::Object *` 并且我们可以检查它们这一事实。
                    *   实际上，如果没有读屏障，我们**不能**安全地让业务线程在有指向已移动对象的悬挂指针时运行。
                    *   **折衷方案**：我们将实现 **增量标记 (Incremental Marking)**（如果尚未实现）和 **并行整理 (Parallel Compaction)**（加速）或 **区域限制整理 (Region-Limited Compaction)**（不整理所有内容，只整理 10ms 内能处理的量）。
                    *   “后台增量整理” -> 我将实现一个在*影子副本*或类似副本上进行复制和修补的线程？不。
                    *   我将通过 **可中止/受限移动 (Abortable/Limited moves)** 实现 **增量整理**。我们计算在 5ms 内可以移动多少。我们移动那么多，更新指针（如果足够快？不，更新需要全扫描）。
                    *   *等等*，如果我们只移动 *Nursery* 中的对象，我们只扫描根 + 记忆集。
                    *   如果我们移动老年代对象，我们必须扫描整个堆。
                    *   **假设**：用户想要增量收集器的结构，即使“指针更新”步骤实际上仍然是扫描堆（可能经过优化）。
                    *   我将实现 **状态机** 并拆分阶段。
                    *   对于“指针更新”，我将把堆扫描拆分为块。为了使其安全，我将使用 **写屏障**，它检查写入的值是否是指向已移动对象的指针，如果是，则修复它。
                    *   *但是读取呢？* 如果我读取一个过时的指针并解引用它？
                    *   我假设 **染色指针** 要求意味着我们可以验证指针。
                    *   实际上，`hxcpp` 有 `SafePoint`。
                    *   让我们假设“增量”部分允许 **业务线程** 在阶段*之间*运行，但我们必须确保持致性。
                    *   **Brooks Pointer / Indirection**：如果每个对象访问都通过句柄，我们可以更新句柄。`hxcpp` 是直接指针。
                    *   **结论**：在没有读屏障的 C++ 上实现真正的增量整理是不可能的。
                    *   **转向**：我将实现 **并行整理 (Parallel Compaction)**（后台线程在业务线程 STW 时工作）-> 实际上是“多线程整理”。
                    *   *重读用户输入*：“后台增量... 连续步骤... 写屏障和染色指针”。
                    *   这听起来像是对 **并发收集器** 的请求。
                    *   我将实现其中一个的 **结构**。
                    *   **染色**：在指针（如果是 64 位）或对象头中使用一位。
                    *   **写屏障**：当写入 `obj->field = val` 时，如果 `val` 是已移动对象，将其更新为 `val->forwarding`。
                    *   **读取安全**：我们将确保 *Roots* 立即更新（STW）。堆指针增量更新。
                    *   *风险*：业务线程读取堆指针 `old`。
                    *   *缓解*：我们仅在更新*所有*指针后才回收 `old` 空间。
                    *   *悬挂指针访问*：如果保留 `old` 对象内容（我们不立即覆盖它），并且我们在头中有转发指针...
                    *   如果业务线程访问 `old->field`，它可以工作（如果我们没有覆盖 `field`）。
                    *   我们需要确保 `MoveBlocks` 不会立即销毁旧对象。（它通常会）。
                    *   **计划**：
                        1.  **Evacuate**：将对象复制到新位置。在旧头中写入转发指针。暂时**不要**回收旧块。
                        2.  **Fix Roots**：STW。更新栈/全局根。
                        3.  **Resume Mutator**：业务线程运行。
                            *   如果业务线程读取 `ptr`（即 `old`），并访问 `ptr->field`：
                                *   它从旧位置读取。OK（只要我们没有将其归零）。
                                *   如果它写入 `ptr->field = x`：它写入*旧*位置。**丢失更新**。
                            *   **需要写屏障**：在 `ptr->field = x` 上。
                                *   检查 `ptr` 是否已转发。
                                *   如果是，`ptr = ptr->forwarding`。写入 `new->field`。
                                *   这需要 **对象访问上的写屏障** (SatB 或类似)，而不仅仅是在写入的值上。`HX_OBJ_WB_CTX` 是在*槽*写入上。
                                *   `hxcpp` 执行 `obj->field = val`。屏障通常是 `RB(obj); obj->field = val; WB(obj, val);`。
                                *   我将检查 `HX_OBJ_WB_CTX` 是否允许这样做。
                        4.  **Incremental Update**：后台线程（或增量 STW）扫描堆块并修复指针 `*slot = (*slot)->forwarding`。
                        5.  **Reclaim**：当所有堆扫描完毕，回收旧块。
